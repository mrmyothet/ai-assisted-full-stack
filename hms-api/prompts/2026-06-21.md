You are working on the `hms-api` project, which serves as the backend API for the Admin Dashboard.

## Task

Implement a complete email-based authentication system using Better Auth.

## Requirements

### Authentication Features

* Add authentication support for the Admin Dashboard.
* Use **Better Auth** as the authentication framework.
* Support **Login Only**.
* Do NOT implement:

  * User registration/signup
  * Social login providers (Google, GitHub, Facebook, etc.)
  * Passwordless login
  * Magic links

### Login Flow

1. User enters email address.
2. User enters password.
3. Validate email and password.
4. If credentials are valid:

   * Generate a one-time verification code (OTP).
   * Send the OTP to the user's email address.
5. User submits the OTP.
6. Verify the OTP.
7. Create an authenticated session and return the session/token.
8. Reject login if OTP verification fails.

### Email Delivery

* Use the **Resend** library/service for sending OTP emails.
* OTP emails should:

  * Contain only the verification code and basic instructions.
  * Include an expiration time.
* Store Resend API key in environment variables.

### Security Requirements

* Hash passwords securely.
* OTP must:

  * Be randomly generated.
  * Have an expiration time (e.g. 5–10 minutes).
  * Be single-use.
* Rate limit OTP generation and verification attempts.
* Prevent brute-force attacks.
* Return proper error responses for invalid credentials, expired OTPs, and invalid OTPs.

### API Endpoints

Create endpoints similar to:

* POST `/auth/login`

  * Validate email/password
  * Generate and send OTP

* POST `/auth/verify-otp`

  * Verify OTP
  * Create authenticated session

* POST `/auth/logout`

  * Destroy session

* GET `/auth/session`

  * Return current authenticated user/session

### Deliverables

1. Database schema changes.
2. Better Auth configuration.
3. Resend integration.
4. OTP generation and verification logic.
5. API route implementations.
6. Environment variables required.
7. Example request/response payloads.
8. Migration scripts if needed.

### Acceptance Criteria

* Admin user can log in using email and password.
* OTP email is delivered through Resend.
* Session is created only after successful OTP verification.
* No signup endpoint exists.
* No social login exists.
* Authentication follows Better Auth best practices.
* Code is production-ready, secure, and well-structured.

Before implementing, analyze the existing `hms-api` codebase and follow its current architecture, coding standards, dependency injection patterns, folder structure, and error handling conventions.
